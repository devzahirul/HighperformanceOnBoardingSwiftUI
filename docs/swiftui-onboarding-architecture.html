SwiftUI Architecture Guide

High-Performance Onboarding Flows with SwiftUI

Architectural patterns, state management strategies, and performance techniques for building production-grade onboarding experiences.

Core Architectural Patterns

Pattern 01
Coordinator + Router
Decouples navigation logic from views. A dedicated OnboardingCoordinator owns the step sequence, making it trivial to A/B test flows or skip steps based on user state — without mutating individual view logic.
NavigationStack Coordinator Testable
Pattern 02
State Machine (Enum-Driven)
Model each step as an explicit enum case. Transitions are pure functions — OnboardingState → Event → OnboardingState. Impossible states become unrepresentable, and the entire flow is unit-testable without a simulator.
Enum Cases Pure Functions Unit Testable
Pattern 03
MVVM + ObservableObject
Each onboarding screen has a dedicated ViewModel that owns its validation logic. The shared OnboardingStore accumulates answers. Presentation is a pure function of state — ideal for snapshot testing.
@Observable ViewModel Snapshot Tests
Pattern 04
Composable Page Builder
Define each screen via a data model (OnboardingPage). A single generic renderer drives the entire flow. Adding a new step = adding a new config struct. Zero new SwiftUI views required.
Data-Driven Reusable CMS-Friendly
Pattern 05
Lazy Dependency Injection
Heavy services (biometrics, camera permission, analytics) are injected lazily only when their screen enters the hierarchy. Avoids paying the initialization cost upfront. Use @Environment for propagation.
@Environment Lazy Init Protocols
Pattern 06
Persistence-First Resume
Persist each completed step immediately (UserDefaults or SwiftData). On cold launch, restore to the last incomplete step. Users who drop off mid-flow seamlessly continue — critical for long permission-request flows.
SwiftData UserDefaults Resume

State Management Comparison

Approach Complexity Testability Best For Avoid When
@State + Enum Cases Low Medium Simple linear flows (3–5 steps) Branching logic or A/B tests
@Observable ViewModel Medium High Per-screen validation & async work Deeply nested shared state
TCA (The Composable Architecture) High Very High Complex branching, large teams Small apps or solo projects
Coordinator + Router Medium High Reusable onboarding across targets One-off, simple linear flows
NavigationPath (iOS 16+) Low Medium Deep link support into mid-flow Flows with complex presentation logic
💡
Recommended default: Start with @Observable ViewModels per screen + a shared OnboardingCoordinator holding a NavigationPath. Migrate to TCA only when multi-team coordination or extensive side-effect testing is required.

Implementation Examples

OnboardingCoordinator.swift Swift 
// Enum-driven state machine — impossible states unrepresentable
enum OnboardingStep: Hashable, CaseIterable {
    case welcome
    case permissions
    case profile
    case preferences
    case complete
}

@Observable
final class OnboardingCoordinator {
    var path: NavigationPath = .init()
    var collectedData: OnboardingData = .init()

    private let persistence: OnboardingPersistence

    init(persistence: OnboardingPersistence = .live) {
        self.persistence = persistence
        self.path = persistence.restoreNavigationPath()
    }

    func advance(from step: OnboardingStep) {
        let next = nextStep(after: step, data: collectedData)
        persistence.markCompleted(step: step)
        if let next { path.append(next) }
    }

    // Pure function — unit-testable without UIKit/SwiftUI
    func nextStep(after step: OnboardingStep,
                   data: OnboardingData) -> OnboardingStep? {
        switch step {
        case .welcome:     return .permissions
        case .permissions: return data.needsProfile ? .profile : .preferences
        case .profile:      return .preferences
        case .preferences: return .complete
        case .complete:    return nil
        }
    }
}
OnboardingRootView.swift Swift 
// Root view — coordinator owns navigation, views are pure renderers
struct OnboardingRootView: View {
    @State private var coordinator = OnboardingCoordinator()

    var body: some View {
        NavigationStack(path: $coordinator.path) {
            WelcomeView()
                .navigationDestination(for: OnboardingStep.self) { step in
                    stepView(for: step)
                }
        }
        .environment(coordinator)
    }

    @ViewBuilder
    private func stepView(for step: OnboardingStep) -> some View {
        switch step {
        case .permissions:  PermissionsView()
        case .profile:      ProfileSetupView()
        case .preferences: PreferencesView()
        case .complete:    CompletionView()
        default:           EmptyView()
        }
    }
}
ProfileSetupViewModel.swift Swift 
// Per-screen ViewModel — isolated validation, easy to unit test
@Observable
final class ProfileSetupViewModel {
    var name: String = ""
    var avatarData: Data?
    var isLoading: Bool = false
    var validationError: String?

    // Computed — no stored state needed
    var canAdvance: Bool { name.trimmingCharacters(in: .whitespaces).count >= 2 }

    func submit(coordinator: OnboardingCoordinator) async {
        guard canAdvance else {
            validationError = "Name must be at least 2 characters"
            return
        }
        isLoading = true
        defer { isLoading = false }

        coordinator.collectedData.name = name
        coordinator.collectedData.avatarData = avatarData
        await coordinator.advance(from: .profile)
    }
}
✅
Pro tip: Keep ViewModels free of any SwiftUI import. They should compile as plain Swift modules. This enforces the separation of concerns and makes XCTest coverage straightforward without needing @MainActor ceremony everywhere.

Performance Techniques

Rendering
Lazy Asset Loading
Use AsyncImage or pre-decode images on a background actor. Never block the main thread during page transitions. Prefetch the next step's assets during the current step's animation.
Animation
Matched Geometry Effects
Use matchedGeometryEffect sparingly. For hero transitions between steps, prefer .transition(.asymmetric(...)) with custom timing curves over matched geometry to avoid layout recalculation storms.
Memory
View Lifecycle Control
NavigationStack keeps previous views alive in memory. For heavyweight steps (camera, video), use sheet/fullScreenCover so they deallocate on dismiss. Use .onDisappear to cancel subscriptions eagerly.
Startup
Deferred Permission Requests
Never request permissions at app launch. Initialize permission managers only when the relevant onboarding step appears. Use .task modifier with proper cancellation — not .onAppear — for async setup.
Layout
Avoid Expensive Layouts
Onboarding screens often use large full-screen layouts. Prefer fixed-size containers over GeometryReader when possible. GeometryReader triggers a second layout pass — avoid nesting them.
Observation
Granular @Observable Scoping
With @Observable (iOS 17+), only the view that reads a property re-renders when it changes. Split large observable objects into smaller, focused ones. Avoid putting unrelated state in a single observable class.
⚠️
Common pitfall: Placing network calls or permission checks inside body computed property closures. SwiftUI can call body dozens of times. All side effects belong in .task, .onAppear, or dedicated async methods on your ViewModel.
PermissionsView.swift — Task-Based Setup Swift 
struct PermissionsView: View {
    @State private var viewModel = PermissionsViewModel()

    var body: some View {
        PermissionsContent(viewModel: viewModel)
            // .task is cancelled if view disappears — safe by default
            .task { await viewModel.loadCurrentStatus() }
            // Prefetch next step's resources during this step
            .task(priority: .background) {
                await ProfileAssets.prefetch()
            }
    }
}

Production Readiness Checklist

✓

Flow is resumable on cold launch

Persist completed steps. Restore NavigationPath from storage. Test by force-quitting mid-flow.

✓

All async work uses .task, not .onAppear

Ensures proper cooperative cancellation when views disappear. No orphaned tasks.

✓

ViewModels have zero SwiftUI imports

Pure Swift business logic. Covered by XCTest with no host app or simulator required.

✓

Step transitions are testable pure functions

nextStep(after:data:) takes no side effects. Branching logic has 100% unit test coverage.

✓

Analytics events fire at transition, not view appear

SwiftUI views can appear multiple times. Track progress in the coordinator, not individual views.

✓

Dynamic Type and accessibility tested

Run Accessibility Inspector. Onboarding screens often have custom layouts that break at xxxLarge text sizes.

✓

Permission screens handle "denied" state gracefully

Show deep-link to Settings.app. Never block flow on denied permissions — make them skippable.

✓

Back navigation reviewed deliberately

Decide: should users go back? If not, hide the back button explicitly. If yes, restore form state.

✓

Onboarding skippable in tests via launch argument

Add ProcessInfo check: if -skipOnboarding is set, mark all steps complete. Unblocks UI test suites.

✓

Data collected in coordinator, not individual views

Views write to coordinator.collectedData. On completion, coordinator sends one consolidated payload to the backend.

🏁
iOS 17+ bonus: Use @Observable instead of ObservableObject + @Published throughout. It eliminates unnecessary re-renders by tracking only the properties actually read by each view — yielding measurably better scroll and transition performance in multi-step flows.
